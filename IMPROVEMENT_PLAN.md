# Area-Lental 고도화 및 안정화 계획

> 전체 코드베이스 분석 결과를 바탕으로, 6개 섹션으로 나누어 단계적으로 진행합니다.
> 각 섹션은 독립적으로 작업 가능하며, 우선순위 순으로 정렬되어 있습니다.

---

## 섹션 1: 테스트 커버리지 확충

**현황**: pricing.test.ts, schema.test.ts, mask.test.ts — 3개만 존재. API·컴포넌트·통합 테스트 전무.

| # | 작업 | 상세 |
|---|------|------|
| 1-1 | API 라우트 단위 테스트 | POST /api/requests (정상·유효성실패·레이트리밋), DELETE /api/admin/requests (인증·상태검증) |
| 1-2 | 비즈니스 로직 테스트 보강 | bundle.ts (묶음 분석), operating.ts (운영시간 판정), gallery.ts (갤러리 기간 계산) |
| 1-3 | 유틸리티 테스트 | datetime.ts, publicLinkToken.ts (매직링크 생성·검증), auditLog.ts |
| 1-4 | 이메일 템플릿 테스트 | mail.ts 렌더링 결과 스냅샷, 변수 치환 정상 확인 |
| 1-5 | tsconfig에서 \_\_tests\_\_ exclude 제거 | 현재 테스트 파일이 타입체크에서 제외되어 타입 오류 미감지 위험 |

## 섹션 2: 접근성(A11y) 및 UX 개선

**현황**: 기본 시맨틱 HTML 사용 중이나 ARIA 속성·키보드 네비게이션·포커스 관리 미흡.

| # | 작업 | 상세 |
|---|------|------|
| 2-1 | 모달 포커스 트랩 | PledgeModal, PrivacyModal — 열릴 때 포커스 이동, ESC 닫기, 닫힐 때 트리거 요소 복귀 |
| 2-2 | 폼 ARIA 라벨링 | ApplyClient의 복합 입력(생년월일 3개 셀렉트, 추가 날짜 등)에 aria-label/aria-describedby |
| 2-3 | 테이블 접근성 | RequestTable에 scope="col"/scope="row", caption 추가 |
| 2-4 | 달력 키보드 내비게이션 | SpaceBooking·SpaceBookingGalleryRange — 방향키 이동, Enter 선택 |
| 2-5 | 스킵 링크 추가 | 전체 레이아웃에 "본문 바로가기" 링크 |
| 2-6 | 색상 대비 점검 | 상태 뱃지·disabled 텍스트 등 WCAG AA 기준 충족 확인 |

**현황**: 서버리스 환경 기본 최적화 완료. 대량 데이터 시 병목 가능성 있음.

## 섹션 4: 보안 강화

**현황**: 이미 높은 수준(CSP, Rate Limit, 타이밍 공격 방어 등). 잔여 개선 사항 해결.

| # | 작업 | 상세 |
|---|------|------|
| 4-1 | CSP unsafe-inline 제거 | 인라인 스타일에 nonce 기반 허용으로 전환 |
| 4-2 | 파일 업로드 매직바이트 검증 | MIME 타입 외 파일 시그니처(magic bytes) 검증 추가 |
| 4-3 | API 에러 응답 구조화 | 모든 API에 일관된 에러 코드 체계 적용 (현재 일부만 VALIDATION_ERROR 등 사용) |
| 4-4 | npm audit 정리 | 의존성 취약점 스캔 및 패치 |
| 4-5 | 헬스체크 엔드포인트 | /api/health — DB 연결·환경변수 상태 확인용 |

## 섹션 5: 코드 품질 및 일관성

**현황**: 전반적으로 양호하나 일부 패턴 불일치, 에러 처리 미세 차이 존재.

| # | 작업 | 상세 |
|---|------|------|
| 5-1 | API 라우트 에러 핸들링 통일 | try-catch 패턴, 로깅 포맷, 응답 구조를 공통 유틸로 추출 |
| 5-2 | 데이터베이스 레이어 정리 | database.ts의 lazy import 패턴 → 첫 import 후 캐시, 타입 추론 개선 |
| 5-3 | 컴포넌트 Props 타입 export | 재사용 컴포넌트의 Props 타입을 명시적으로 export (현재 일부 인라인) |
| 5-4 | 공통 상수 정리 | 매직 넘버(200개 삭제 제한, 30분 세션 등) → config.ts로 집중화 |
| 5-5 | ESLint 규칙 강화 | no-console(warn), exhaustive-deps, prefer-const 등 추가 |

## 섹션 6: 운영 안정성

**현황**: Vercel 배포 기본 설정. 모니터링·백업·장애 대응 체계 미비.

| # | 작업 | 상세 |
|---|------|------|
| 6-1 | 구조화된 로깅 강화 | 요청별 traceId 부여, API 응답시간 측정 |
| 6-2 | Google Sheets 장애 대응 | API 호출 실패 시 재시도 로직(exponential backoff) + 사용자 안내 메시지 |
| 6-3 | 이메일 발송 재시도 | SMTP 실패 시 3회 재시도 + 실패 기록 |
| 6-4 | 환경변수 검증 강화 | 서버 시작 시 필수 환경변수 누락 → 명확한 에러 메시지와 함께 즉시 종료 |
| 6-5 | DB 백업 가이드 | Google Sheets 자동 백업 스크립트 또는 정기 export 방안 |

## 권장 작업 순서

```
섹션 1 (테스트) → 섹션 5 (코드 품질) → 섹션 4 (보안) → 섹션 2 (접근성) → 섹션 3 (성능) → 섹션 6 (운영)
```

**이유:**
- **섹션 1 먼저** — 테스트가 있어야 이후 리팩토링/개선 시 회귀 방지 가능
- **섹션 5 다음** — 코드 일관성 확보 후 보안·접근성 작업이 더 수월
- **섹션 6 마지막** — 운영 안정성은 코드 품질이 확보된 후 적용
